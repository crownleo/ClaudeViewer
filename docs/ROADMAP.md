# 🗺️ ClaudeViewer 路线图 Roadmap

> 把零散的 Claude 对话，逐步变成**可管理、可复用的本地知识资产**。  
> 本文档代表**方向而非承诺**，优先级会根据反馈调整。  
> 分两档：**承诺方向**（阶段 1–3）与**远期探索**（阶段 4）。  
> 最后更新：2026 年 7 月 · 当前版本 **v5.1**

---

## 🧭 设计原则（一切规划的底线）

任何新功能都必须遵守，**违背原则的需求一律不做**：

1. **本地优先（Local-first）** — 数据永远只在你的浏览器里处理，不经过任何服务器。
2. **单文件优先 + 可选伴侣** — 核心查看器**永远是一个 `claude_viewer.html`**，双击即用、零安装。少数重型功能（如语义搜索、本地摘要）做成**可选升级**（PWA / 本地小工具），不需要就不装；**绝不因为加功能而牺牲"双击即用"和隐私**。
3. **隐私至上（Privacy）** — 不上传、不追踪、不埋点。你的对话是你的。

---

## 🪜 四阶段价值阶梯

| 阶段 | 定位 | 一句话 | 状态 |
|------|------|--------|------|
| **1. 数据查看器** | 打得开、看得懂 | 让你成功打开、搜索、导出自己的 Claude 数据 | ✅ 基本完成 |
| **2. 本地归档器** | 长期管理 | 把历史对话变成可管理的个人档案 | 🔜 进行中 |
| **3. 本地知识库** | 知识复用 | 把零散对话变成可搜索、可复用、可整理的知识资产 | 🌿 目标 |
| **4. AI 记忆中枢** | 多来源记忆 | 管理你与各类 AI 的历史交互 | 🔭 远期探索 · 未承诺 |

## ✅ 阶段 1：Claude 数据查看器（基本完成）

**核心价值**

| 功能 | 解决的问题 |
|------|-----------|
| 导入 Claude 导出文件 | 数据打不开 |
| 查看对话 | 阅读困难 |
| 搜索历史内容 | 找回以前问过的问题 |
| 导出 Markdown / PDF | 方便保存和二次整理 |
| 统计分析 | 看到自己的使用情况 |

**目标**：让用户成功打开、搜索、导出自己的 Claude 数据。 → **已达成**，持续小步打磨：

- [ ] 英文界面 / 国际化（i18n）
- [ ] 键盘快捷键（`⌘/Ctrl+F` 聚焦搜索、`↑↓` 切换结果、`Esc` 关闭面板）
- [ ] 移动端 / 响应式优化

---

## 🔜 阶段 2：Claude 本地归档器（进行中）

不只是"打开一次"，而是**长期管理**。

**目标**：把 Claude 历史对话变成可管理的个人档案。

**头号功能（优先做）**

- [ ] **增量导入 + 自动去重** — 新导出的数据不再重复导入，自动识别重复对话。这是"查看器 → 归档器"的枢纽。
  > 已有可复用基础：`finalizeData()` 已用 `uuid` 对项目去重，扩展到对话即可。

**其余功能**

- [ ] **标签增强** — 给对话打标签（如"编程""英语""论文""简历"），支持批量、重命名
- [ ] **收藏分组** — 标记并归类重要对话
- [ ] **项目 / 主题分类** — 按 Claude Projects 或主题组织
- [ ] **时间轴** — 按日期查看历史
- [ ] **高级筛选** — 按时间、关键词、附件、模型、项目组合筛选

**🏗 架构注记（趁早埋接缝）**  
本阶段顺手引入**统一对话数据模型 + 可插拔导入层**。现状是解析代码写死了 Claude 导出格式（`classifyJson` 硬编码文件名、`chat_messages` / `content[].type` 直接贯穿全程）。趁数据量还小把归一化层抽出来，阶段 4 的多来源才不至于推倒重写。**这是一次性的低成本投资，回报在后面。**

---

## 🌿 阶段 3：本地知识库（目标）

核心不是"存对话"，而是：**把零散的 Claude 对话，变成可搜索、可复用、可整理的知识资产。**

按架构分两条子轨 —— 先做能塞进单文件的，AI 能力走可选伴侣。

**(A) 单文件内即可实现（低风险，优先）**

- [ ] **Markdown 知识库导出** — 导出到 Obsidian / Notion / Typora（纯格式化，无 AI，无隐私风险，直接接入你已在用的工具）
- [ ] **自动标题优化** — 用规则 / 启发式给历史对话生成更清楚的标题
- [ ] **双链 / 引用** — 在对话之间建立关联
- [ ] **全文索引加速** — 浏览器本地索引，让所有历史对话搜索更快

**(B) 需要"可选伴侣"的 AI 能力（opt-in，绝不必需）**

- [ ] **本地摘要** — 给长对话生成摘要
- [ ] **主题聚类** — 自动识别"编程 / 英语 / 面试 / 论文 / 工具"等主题
- [ ] **知识卡片提取** — 从对话中提取结论、步骤、代码、清单
- [ ] **语义搜索** — 不记得原词，也能找到相关内容

> **🔒 隐私边界不能破。** 本地、隐私、不上传是本项目最大的卖点。以上 AI 能力一旦落地，**只会**通过以下方式实现：
> - 本地模型（Ollama / LM Studio）
> - 浏览器本地索引
> - 用户明确选择的自带 API Key（BYO Key）
> - **默认不上传任何数据**
>
> **绝不偷偷接云端 API** —— 那会摧毁项目赖以立足的信任。

## 🔭 阶段 4：个人 AI 记忆中枢（远期探索 · 未承诺）

更远一点，ClaudeViewer 也许能成为 **Personal AI Conversation Memory Hub** —— 管理你与各类 AI 的历史交互。

**重点不是"支持所有平台"，而是"数据模型做好、导入器可插拔"**（正是阶段 2 埋下的接缝）。只要 Claude 建模得当，新增一个来源就只是加一个导入器。

| 数据源 | 建议 |
|--------|------|
| **Claude** | 当前核心 |
| ChatGPT 导出 | 中期可考虑 |
| Gemini 导出 | 中期可考虑 |
| Markdown 文件夹 | 很适合 |
| Obsidian Vault | 很适合 |
| Cursor / Codex / Claude Code 历史 | 有潜力 |
| 微信聊天记录 | ❌ 不做（隐私与合规太复杂） |
| 邮件 / 网盘 | ❌ 暂不考虑（太重） |

> **纪律**：先把 Claude 做深（阶段 1–3），再谈多来源。一上来就"大一统 AI 聊天记录管理器"，很容易变成烂尾大杂烩。

---

## 🚫 明确不做（守住产品哲学）

- ❌ 将数据上传到任何服务器 / 云端
- ❌ 账号系统、登录、云同步
- ❌ 任何需要后端服务才能运行的功能
- ❌ 用户行为追踪、分析埋点、广告
- ❌ 微信 / 邮件 / 网盘导入（隐私与合规负担过重）
- ❌ **偷偷接云端 API**（会直接摧毁项目信任）
